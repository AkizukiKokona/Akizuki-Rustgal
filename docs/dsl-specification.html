1Overview

.akrs is the scripting language of the Akizuki*Rustgal visual-novel engine. It is a compact, line-oriented DSL whose grammar favors symbolic markers over English keywords. A script author writes scenes, dialogue, choices, and branching flow in a natural, readable form — while the engine guarantees correctness before the game ever runs.

Unlike Python-based engines such as Ren'Py, .akrs deliberately avoids constructs like label, jump, call, menu, with (as a keyword), Python-style indentation semantics, and variable interpolation. Instead, structure is conveyed by distinctive glyphs (#, ->, ?, |, @, +, $) that are visually unambiguous and require no embedded programming language.

»Compilation Pipeline

A .akrs file passes through the same toolchain whether it is compiled ahead-of-time or reloaded at runtime:

source (.akrs)→ Lexer→ Parser→ AST→ Semantic Checker→ VM Bytecode→ Runtime Events

The checker is the shared backbone: it runs identically inside a Rust proc macro (compile time) and inside the runtime (hot reload), so an author always gets the same diagnostics in the editor and in the live game.

if · else · end · wait and · or · not # · -> · => · <= ? · | · @ · + · $

2Design Principles

The language was shaped by a handful of guiding ideas:

☞

For non-programmers

Script authors think in scenes and dialogue, not in functions or classes. Syntax stays close to natural language.

≡

Symbolic, not keyword-heavy

Distinctive glyphs mark structure so blocks are obvious at a glance — without a Ren'Py-style keyword vocabulary.

[]

Explicit block delimiters

Blocks open and close with explicit markers (?…?, if…end). Indentation is visual only — never semantic.

{ }

No embedded languages

There is no inline Python, Ruby, or shell. Everything is statically parseable and statically checkable.

✓

Compile-time verified

Rust proc macros reject broken scripts at build time: dangling sections, type mismatches, missing resources.

↻

Reloadable

The same checker runs at runtime, so scripts can be edited and reloaded without restarting the engine.

Not Ren'Py

The following Ren'Py concepts do not exist in .akrs: label, jump, call/return, menu, with as a statement keyword, Python blocks, indentation-based blocks, and [variable] interpolation. Their equivalents use symbolic markers and are documented in §4.

3Quick Start

A valid .akrs script is simply one or more sections. The first section is the entry point of the story. Here is the smallest meaningful script:

# Start

@bg school_entrance with fade
+ Aki enters from left with dissolve

Aki: "Hello! Welcome to the story."

$met_aki = true

? "What do you say?"
| "It's nice to meet you."
    Aki (happy): "I'm glad!"
    -> GoodEnding
| "Sorry, I'm busy."
    Aki (sad): "Oh... okay."
    -> GoodEnding
?

# GoodEnding

"And so the day began."
~~

Reading this top-to-bottom you can already see every major feature: a section header (#), a stage command (@bg), a stage direction (+), dialogue with a pose, a variable assignment ($), a choice block (?…? with | options), flow navigation (->), and an explicit story end (~~).

Tip

Indentation in the example above is purely for human readability. The parser ignores it entirely — you may indent however you like, or not at all.

4Syntax Reference

This section documents every construct in the language. Each subsection lists the syntax signature, a description, and an example.

4.1Sections

# SectionName # (closing # is optional)

Sections are the basic organizational unit of a script — the analogue of a "label" or "scene". A section begins with a line containing a hash mark #, the section name, and an optional closing #. The name is an identifier (letters, digits, underscore).

  • The first section in the file is the story's entry point.
  • Section names must be unique; duplicates are a compile-time error.
  • Section names may be referenced before they are defined (forward references are allowed).
  • Every node must live inside a section. Nodes appearing before the first section header are rejected.
# Prologue
-- nodes go here...

# Act1
-- ...more nodes...

4.2Dialogue

Character (pose) : "text"

A spoken line. The speaker name (an identifier) is followed by a colon and a quoted string. An optional pose may be given in parentheses between the name and the colon — this is typically mapped by the renderer to a sprite expression or animation state.

Aki: "Hey! You're here early today."
Aki (happy): "I was worried you might not come."
Narrator (serious): "The wind picked up."

4.3Narration

"narration text"

A standalone quoted string on its own line is narration — text shown without a speaker name box. This distinguishes it from dialogue (which requires a Name: prefix).

"Cherry blossoms drift through the afternoon air."
"They parted ways as the sun set."
Note

If a quoted string is preceded by Name: it is dialogue; if it appears alone on a line it is narration. The parser decides based on whether an identifier-and-colon prefix is present.

4.4Stage Commands

@ command args… with transition

The @ prefix introduces a stage command that affects the scene rather than the characters' speech. The command name is an identifier (e.g. bg, music, sound) followed by zero or more arguments (identifiers or quoted strings). An optional with transition suffix applies a visual/audio transition.

@bg school_entrance with fade
@bg cafe with dissolve
@music gentle_rain
@sound doorbell
@bg "evening sky" with fade_white
CommandPurposeTypical args
bgChange the background imagebackground name
musicStart or swap background musicmusic file
soundPlay a one-shot sound effectsound effect
Resource check

When an assets directory is registered, the bg, music, and sound arguments are checked against known resources at compile time. A missing resource is reported as a warning. See §7.

4.5Stage Directions

+ Character enters|exits from|to|at position with transition

The + prefix introduces a stage direction that moves a character sprite on or off stage. The character name is followed by a verb (enters/exits), an optional position clause, and an optional transition.

+ Aki enters from left with dissolve
+ Aki exits to right with fade
+ Aki enters at center
+ Hero enters from 0.25 with slide_right
enters · enter
Bring the character onto the stage.
exits · exit · leaves · leave
Remove the character from the stage.
from · to · at
Introduces a position. For enters, from or at is conventional; for exits, to is conventional. All three keywords are accepted in either direction.
with transition
Applies a transition effect to the entrance/exit.

Positions and transitions are listed in §6.

Exit shorthand — the - prefix

- Character exits with transition

As a convenience, a character may be removed from the stage using the - prefix instead of writing + Character exits. The exits verb is optional with - (it is implied). An optional with transition suffix applies a transition effect to the departure. After a character exits, the engine automatically re-lays-out the remaining sprites (see §6.3).

- Aki with fade
- Aki exits with dissolve
- Hero
Equivalence

- Aki is equivalent to + Aki exits. Use whichever reads more naturally in context. The - form is shorter and pairs visually with the + entrance prefix.

4.6Variables

$ varName = | += | -= expr

The $ prefix introduces a variable operation. Variables hold the story state (affection points, flags, counters). Three operators are supported:

OperatorMeaningExample
=Assign (also declares the variable on first use)$affection = 5
+=Add-assign (requires an existing, type-compatible variable)$affection += 1
-=Subtract-assign (requires an existing, type-compatible variable)$health -= 10
 
$met_aki = true
$affection = 1
$name = "Aki"
$pi = 3.14
$affection += 3
$health -= 10
Compile-time

Using += or -= on a variable that was never assigned is an error. The right-hand side type must also be compatible with the variable's inferred type (int and float are mutually compatible; see §5).

4.7Flow Navigation

-> TargetSection

The flow arrow -> performs a one-way transfer to another section. Execution continues at the top of the target section and never returns. Because it is one-way, the visit/call stack is cleared when a flow is taken — any pending => return addresses are discarded.

-> Date
-> GoodEnding
Not jump

-> replaces Ren'Py's jump. The target section must exist — this is verified at compile time with "did you mean?" suggestions for typos.

4.8Visit & Return

=> TargetSection  …  <=

The visit arrow => transfers to another section while remembering where to come back. It pushes a return address onto an internal call stack. The return mark <= pops that address and resumes execution immediately after the originating =>.

# Main
Aki: "Let me check something."
=> PhoneCall
Aki: "Anyway, where were we?"

# PhoneCall
"Ring ring..."
Aki: "Hello?"
<=
Tip

Visits are ideal for shared interludes (a phone call, a flashback) that several sections reuse. Because flow -> clears the call stack, a one-way flow taken inside a visited section will not return — it abandons the visit, which is usually the desired "go somewhere permanent" behavior.

Runtime

Issuing <= with an empty call stack is a runtime error (return without visit). Reaching the end of any section also returns to the caller if the call stack is non-empty; otherwise it ends the story.

4.9Choices

? "prompt"
| "option" if expr
    body…
?

A choice block is opened and closed by ?. Options are introduced with | followed by a quoted label. Each option may carry an optional if expr guard: when the guard is false at runtime, that option is hidden or disabled. The body of an option is the sequence of nodes that execute when the player picks it, and runs until the next | or the closing ?.

? "What do you say?"
| "Yes" if $affection > 5
    Aki: "Great!"
    -> Ending1
| "No"
    Aki: "Oh..."
    -> Ending2
?
  • The prompt string is optional; ? may open the block directly.
  • A guard expression must evaluate to a boolean (see §5); a non-boolean guard is a compile-time error.
  • Option bodies can contain any node: dialogue, commands, variables, nested choices, conditionals, flow, etc.
Not menu

?…? replaces Ren'Py's menu. The same ? glyph opens and closes the block, keeping the syntax symmetrical.

4.10Conditionals

if expr
    body…
else
    body…
end

Conditionals branch on a boolean expression. The if body runs when the expression is truthy; the optional else body runs otherwise. Every if must be closed with an explicit end keyword. There is no indentation-based block structure — the body extends until else or end.

if $affection > 10
    Aki: "You're wonderful!"
else
    Aki: "Let's be friends."
end
Compile-time

The condition expression must be boolean. The checker rejects conditions inferred as a non-boolean type (e.g. a bare string). Integers are truthy/falsy at runtime, but comparisons (>, ==, …) produce clean booleans.

4.11Wait

wait seconds

Pauses execution for the given number of seconds. The argument is a number literal (integer or float). The pause is emitted to the renderer as a Wait event.

wait 2.0
wait 1
wait 0.5

4.12Comments

-- comment to end of line

A double dash -- begins a comment that runs to the end of the line. Comments are ignored by the lexer and may appear anywhere whitespace is allowed.

-- Set the scene
@bg school_entrance with fade  -- nice entrance
$affection = 1  -- starting value

4.13Story End

~~

The double tilde ~~ explicitly ends the story, stopping the virtual machine and emitting a StoryEnd event. Reaching the end of the last executed section also ends the story naturally, so ~~ is most useful when you want to terminate mid-section (for example, right after a final line of dialogue).

Aki: "Thank you for today."
Aki (happy): "Let's do this again sometime."
~~
Implicit end

If a section simply runs out of nodes, the VM ends the story (or returns to a caller if inside a visit). ~~ is therefore an explicit, early terminator.

5.4Value & Type System

At runtime, every value is one of five kinds. The checker infers a compile-time type for each expression to catch mismatches early.

Runtime valueCompile typeTruthiness
Int (i64)intnon-zero is true
Float (f64)floatnon-zero is true
Str (String)stringnon-empty is true
Bool (bool)boolthe value itself
Null—false

Int and float are mutually compatible for arithmetic and assignment (an int may be used where a float is expected and vice-versa). All other type combinations are checked strictly.

6Transitions & Positions

6.1Transitions

Transitions apply to stage commands (@bg … with fade) and stage directions (+ Aki enters … with dissolve). Each has an engine-defined default duration.

NameDefault durationDescription
fade0.6 sGeneral cross-fade.
fade_black0.8 sFade through black.
fade_white0.8 sFade through white.
slide_left0.5 sSlide in/out toward the left.
slide_right0.5 sSlide toward the right.
slide_up0.5 sSlide upward.
slide_down0.5 sSlide downward.
dissolve0.8 sDissolve between sprites/scenes.
wipe_left0.5 sWipe toward the left.

Using a name not in this list is a parse-time error that lists the valid transitions.

6.2Positions

Positions place a sprite horizontally on stage. They are given after from, to, or at in a stage direction. The three named positions correspond to the variants of the engine's Position enum.

6.3Multi-Character Auto-Layout

When multiple character sprites share the stage, the engine automatically computes their horizontal positions so they are evenly distributed and never overlap. The author does not need to specify positions manually for the common cases — the layout is recalculated every time a character enters or exits.

EventResulting layout
Single character entersCentered at 0.5 (Center).
Second character entersFirst moves to 0.25 (Left), second placed at 0.75 (Right).
A character exits — 1 remainingThe remaining sprite is re-centered at 0.5 (Center).
A character exits — 0 remainingThe stage is cleared; no sprites are displayed.

The Position enum used internally by the engine defines exactly three anchor values:

Variantx-fraction
Left0.25
Center0.5
Right0.75
 
+ Aki enters with dissolve        -- Aki alone: centered (0.5)
+ Hero enters with dissolve       -- Aki -> left (0.25), Hero -> right (0.75)
- Hero exits with fade            -- Aki re-centered (0.5)
- Aki exits with fade             -- stage cleared
Auto-layout

An explicit position clause (at center, from 0.25) overrides the auto-layout for that entrance only. When a character later exits (via - or + … exits), the remaining sprites are re-laid-out according to the rules above.

7Compile-Time Checks

The semantic checker runs inside a Rust proc macro at build time (and again at runtime during hot reload, using the identical code). It performs a two-pass analysis: first collecting all section names, then walking every node to validate references, types, and resources.

7.1What is checked

  1. Section reference validity. Every -> and => target must name a section that exists. A missing target is an error, accompanied by a "did you mean?" suggestion computed via edit distance.
  2. Duplicate section names. Each section name must be unique. A duplicate reports the second definition as an error and notes where the first was defined.
  3. Entry section. The script must define at least one section; an empty program produces a warning.
  4. Variable type matching. += and -= require a previously declared variable whose type is compatible with the right-hand side (int/float compatible; otherwise strict equality).
  5. Undefined variable detection. Reading a variable (in +=/=-compound, or inside any expression) that was never assigned is an error.
  6. Resource reference integrity. When an assets directory is registered, the first argument of @bg, @music, and @sound is checked against known resources. A miss is a warning with a hint to add the asset.
  7. Condition types. if conditions and choice if expr guards must be boolean; a non-boolean inferred type is an error.
  8. Expression type inference. Arithmetic, comparison, and logical operators are type-checked. Incompatible operands (e.g. adding an int to a string) are reported.

7.2Examples of caught problems

-- 1) Dangling flow target
-> Datte        -- error: undefined section 'Datte'
                 -- hint: did you mean 'Date'?

-- 2) Compound-assign on an undeclared variable
$score += 5     -- error: undefined variable 'score'
                 -- hint: declare with $score = ... first

-- 3) Type mismatch on compound-assign
$name = "Aki"
$name += 1      -- error: type mismatch: cannot apply += to name (string) and int

-- 4) Non-boolean condition
if "hello"      -- error: if condition must be bool, found string
    Aki: "..."
end
Severity

Diagnostics carry one of three severities: Error (blocks compilation), Warning (e.g. a missing resource), and Note (supplementary context, such as the first definition of a duplicated section).

8Error Reporting

Diagnostics are produced in the style of the Rust compiler (rustc) using codespan-reporting. Each message points to a precise line:column range in the .akrs source, renders the offending line with a caret underline, and may include a hint: note.

error: undefined section: 'Datte' --> story.akrs:12:5 | 12 | -> Datte | ^^^^^ | = hint: did you mean 'Date'?

8.1"Did you mean?" suggestions

For section-reference errors, the checker compares the misspelled target against all known section names using Levenshtein edit distance. If a candidate is within a small distance (3 edits), it is offered as a suggestion. Otherwise the hint advises defining the section with # Name.

8.2Error recovery

The parser synchronizes after an error so that multiple problems can be reported in a single pass. It recovers at section boundaries and at block-closing markers (?, end), then continues parsing the rest of the script.

8.3Diagnostic stages

StageWhenTypical messages
LexerTokenizingUnexpected character, unterminated string, stray ~.
ParserBuilding the ASTUnexpected token, missing :/end/?, unknown transition/position.
CheckerSemantic analysisUndefined section/variable, type mismatch, non-boolean condition, missing resource.

9Hot Reload

.akrs scripts can be modified and reloaded at runtime without restarting the engine. This works because the AST and the checker are serializable and shared between the compile-time proc macro and the runtime.

  • Same checker, two contexts. The semantic checker is plain Rust (no proc-macro types), so it runs identically when embedded at build time and when invoked live. An author sees consistent diagnostics in both places.
  • Re-validation on reload. When a script changes, it is re-lexed, re-parsed, and re-checked before being swapped in. Reload is refused if errors are present, protecting the running game from a broken script.
  • Serializable VM state. The virtual machine's state — current section, instruction pointer, variables, and the visit call stack — is serializable. This enables save/load and lets a reloaded script resume from a compatible position.
  • Event-driven execution. The VM emits discrete events (dialogue, narration, command, direction, choice, wait, story-end) that the renderer consumes, so a content swap does not require engine restart.
Save / Load

Because VM state is serializable, a game can snapshot progress at any point and restore it later — including across a script reload, as long as the section/instruction structure remains compatible.

10Complete Example

The following script exercises nearly every feature of the language: sections, dialogue with poses, narration, stage commands and directions, variables, a choice with a guarded option, conditionals, flow navigation, and explicit story ends. It is a complete, runnable story.

# Prologue

-- Set the scene
@bg school_entrance with fade
+ Aki enters from left with dissolve

"Cherry blossoms drift through the afternoon air."

Aki: "Hey! You're here early today."
Aki (happy): "I was worried you might not come."

$met_aki = true
$affection = 1

? "How do you respond?"
| "I couldn't wait to see you."
    Aki (surprised): "R-really?"
    $affection += 3
    -> Date
| "I had nothing better to do."
    Aki (sad): "Oh... I see."
    $affection -= 1
    -> Walk
?

# Date

@bg cafe with dissolve
+ Aki enters at center

Aki: "This cafe has the best cake."
Aki (happy): "I'm glad we came here together."

if $affection > 3
    Aki: "I... I really like spending time with you."
    -> GoodEnding
else
    Aki: "Let's just enjoy the cake."
    -> NeutralEnding
end

# Walk

@bg park with fade

Aki: "The park is nice this time of year."

if $affection > 0
    -> GoodEnding
else
    -> NeutralEnding
end

# GoodEnding

@bg sunset with fade_white

"And so, under the painted sky, a new chapter began."

Aki: "Thank you for today."
Aki (happy): "Let's do this again sometime."

~~

# NeutralEnding

@bg evening with fade

"They parted ways as the sun set."

Aki: "See you around, I guess."

~~
Trace

Starting in Prologue, the player's choice routes to either Date or Walk. Both evaluate $affection with an if/else and flow to GoodEnding or NeutralEnding, each terminated explicitly with ~~. Notice that affection starts at 1 and is adjusted by the choice — so the guarded branch through Date can reach the good ending only if the affectionate option was picked.

11Grammar Reference

The complete context-free grammar of .akrs, expressed in extended Backus–Naur form. Terminals are quoted; { } denotes zero-or-more, [ ] denotes optional, and | denotes alternation. This grammar mirrors the recursive-descent parser exactly.

program        ::= section { section }

section        ::= "#" IDENT [ "#" ] { node }

node           ::= dialogue
                 | narration
                 | command
                 | direction
                 | varop
                 | choice
                 | conditional
                 | flow
                 | visit
                 | return
                 | wait
                 | storyend

dialogue       ::= IDENT [ "(" IDENT ")" ] ":" STRING
narration      ::= STRING

command        ::= "@" IDENT { arg } [ "with" IDENT ]
arg            ::= IDENT | STRING

direction      ::= "+" IDENT dirverb [ posclause ] [ "with" IDENT ]
                 | "-" IDENT [ dirverb ] [ "with" IDENT ]
dirverb        ::= "enters" | "enter" | "exits" | "exit" | "leaves" | "leave"
posclause      ::= ("from" | "to" | "at") position
position       ::= "left" | "center" | "centre" | "right" | FLOAT

varop          ::= "$" IDENT assignop expr
assignop       ::= "=" | "+=" | "-="

choice         ::= "?" [ STRING ] { option } "?"
option         ::= "|" STRING [ "if" expr ] { node }

conditional    ::= "if" expr { node } [ "else" { node } ] "end"

flow           ::= "->" IDENT
visit          ::= "=>" IDENT
return         ::= "<="
wait           ::= "wait" ( INTEGER | FLOAT )
storyend       ::= "~~"

/* ---- Expressions (precedence climbing) ---- */

expr           ::= or_expr
or_expr        ::= and_expr { "or" and_expr }
and_expr       ::= not_expr { "and" not_expr }
not_expr       ::= "not" not_expr | comparison
comparison     ::= additive { compop additive }
compop         ::= "==" | "!=" | "<" | ">" | "<=" | ">="
additive       ::= multiplicative { ("+" | "-") multiplicative }
multiplicative ::= unary { ("*" | "/") unary }
unary          ::= "-" unary | primary
primary        ::= INTEGER
                 | FLOAT
                 | STRING
                 | "true"
                 | "false"
                 | "$" IDENT
                 | "(" expr ")"

/* ---- Lexical ---- */

IDENT          ::= [A-Za-z_] [A-Za-z0-9_]*
INTEGER        ::= [0-9]+
FLOAT          ::= [0-9]+ "." [0-9]+
STRING         ::= '"' ... '"' | "'" ... "'"
COMMENT        ::= "--" [^\n]*
Indentation

Whitespace and indentation are not part of this grammar. Newlines separate statements but carry no nesting meaning — blocks are delimited solely by their explicit markers.
